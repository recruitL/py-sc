# AGENTS.md

本仓库 `py-sc` 是一个面向 Python 数值计算的课程型 / 书籍型代码库。

它不是单纯的代码仓库，而是一个长期维护的教学与自学项目。每一章应同时包含数学理论、数值算法、Python 实现、Jupyter Notebook 示例、图像展示、实验内容、小结和参考文献。

## 1. 总体定位

本仓库面向数值计算、科学计算和物理计算中的基础方法。

写作风格应接近“可运行的电子讲义”：

* 不是只给代码；
* 不是只给公式；
* 不是百科式堆砌；
* 需要有明确的数学动机、算法结构、数值实验和代码实现。

每一章都应服务于两个目标：

1. 作为课程内容，可以被学生按顺序阅读；
2. 作为个人学习仓库，可以长期扩展和维护。

## 2. 工作原则

修改仓库之前，必须先阅读现有结构，包括但不限于：

* `README.md`
* 章节目录
* notebook 命名方式
* 代码目录
* 文档目录
* 参考文献目录
* `.gitignore`
* 已有个人主页链接或项目介绍

不要在未理解现有结构前大规模重构。优先保持仓库已有风格；如果确实需要调整结构，必须说明理由，并尽量做到局部、可回退、低破坏。

## 3. 章节组织原则

每一章建议包含：

* chapter overview / README
* theory-oriented notebooks
* algorithm-oriented notebooks
* experiment notebooks
* reference notes

每一章至少应有一个统一入口，用来说明：

* 本章主题；
* 阅读顺序；
* 主要数学问题；
* 主要算法；
* notebook 列表；
* 如何运行；
* 本章参考资料。

## 4. Notebook 写作规范

Jupyter Notebook 是本仓库的主要载体。每个 notebook 应尽量包含：

1. 问题背景；
2. 数学定义；
3. 关键公式；
4. 算法思想；
5. Python 实现；
6. 数值实验；
7. 图像展示；
8. 方法局限；
9. 小结。

Notebook 中的代码应以教学清晰为主，不追求过度工程化。允许使用 `numpy`、`matplotlib`、`scipy`，但核心算法应尽量给出手写实现。`scipy` 可以用于对照、验证和展示标准库接口。

图像应清楚说明物理或数学含义，不要只画图不解释。Notebook 输出应适度，不要提交巨大的无意义输出。

## 5. 理论写作规范

理论部分应做到：

* 给出基本定义；
* 说明算法从哪里来；
* 给出关键公式推导；
* 解释误差来源；
* 说明适用条件；
* 说明失败情形；
* 说明与后续章节的联系。

对于重要公式，应优先使用 LaTeX。不要只写最终公式，需要写清楚公式的意义和它在算法中的作用。

## 6. 代码规范

代码应满足：

* 可读；
* 可运行；
* 适合教学；
* 函数命名清楚；
* 变量名尽量反映数学含义；
* 关键步骤有简短注释。

核心算法函数应尽量独立，例如：

* `lagrange_interpolate(...)`
* `newton_divided_differences(...)`
* `cubic_spline_coefficients(...)`
* `chebyshev_nodes(...)`

不要求做成复杂 Python package，除非现有仓库已经采用 package 结构。本仓库当前采用轻量 package 结构，公共算法放在 `src/py_sc/`，章节示例放在 `chapters/<chapter>/`。

## 7. 实验规范

每章应设计若干数值实验，用来说明：

* 算法是否正确；
* 误差如何变化；
* 参数如何影响结果；
* 方法在哪些情况下失败；
* 不同算法之间的差异。

实验应尽量配图，并给出文字解释。不要只输出数字表格。

## 8. 参考文献规范

每章应维护参考文献，可以使用 Markdown、BibTeX 或现有仓库中的参考文献格式。

参考文献应优先包括：

* 数值分析经典教材；
* 科学计算教材；
* Python 科学计算资料；
* 与该章主题直接相关的专业资料。

引用不要求过度完整，但要避免完全没有来源。

## 10. Git 规则

修改完成后必须执行：

```bash
git status
git diff --stat
```

如果可行，应运行必要的代码检查或 notebook 检查。

默认不要执行：

```bash
git push
```

如果用户在当前对话中明确要求“推送”“push”“同步到 GitHub”或“同步到云端”，则应以用户的当前明确指令为准，可以执行 `git push`。发生冲突时，用户当前明确指令优先于本文件中的默认不 push 规则。

不要使用 `sudo`。不要安装系统级依赖。

如果需要新增依赖，优先说明理由，并尽量只写入项目级依赖文件。

新增章节时，不要破坏已有章节。
