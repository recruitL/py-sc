# AGENTS.md

本仓库 `py-sc` 是一个面向 Python 数值计算的课程型 / 书籍型代码库。

它不是单纯的代码仓库，而是一个长期维护的教学与自学项目。每一章应同时包含数学理论、数值算法、Python 实现、Jupyter Notebook 示例、图像展示、实验内容、小结和参考文献。

## 1. 总体定位

本仓库面向数值计算、科学计算和物理计算中的基础方法。

写作风格应接近“可运行的电子讲义”：

* 不是只给代码；
* 不是只给公式；
* 不是百科式堆砌；
* 需要有明确的数学动机、算法结构、数值实验和代码实现。

每一章都应服务于两个目标：

1. 作为课程内容，可以被学生按顺序阅读；
2. 作为个人学习仓库，可以长期扩展和维护。

## 2. 工作原则

修改仓库之前，必须先阅读现有结构，包括但不限于：

* `README.md`
* 章节目录
* notebook 命名方式
* 代码目录
* 文档目录
* 参考文献目录
* `.gitignore`
* 已有个人主页链接或项目介绍

不要在未理解现有结构前大规模重构。优先保持仓库已有风格；如果确实需要调整结构，必须说明理由，并尽量做到局部、可回退、低破坏。

## 3. 语言规范

本仓库的主要读者是中文读者，因此章节内容、Notebook 说明文字、README、理论推导、实验解释和小结应以中文为主。

具体要求：

1. 章节标题、Notebook 标题、正文说明、理论解释、算法说明、实验分析、小结和参考文献说明应使用中文。
2. 数学术语优先使用中文表述，但允许在第一次出现时保留英文原名，例如：
   * 拉格朗日插值（Lagrange interpolation）
   * 牛顿差商（Newton divided differences）
   * 三次样条插值（cubic spline interpolation）
   * 切比雪夫节点（Chebyshev nodes）
   * B 样条（B-spline）
3. 代码中的函数名、变量名、文件名可以使用英文，以保证 Python 代码的可读性和通用性。
4. Notebook 中的 Markdown 说明必须以中文为主，不要生成整段英文讲义。
5. 图像标题、坐标轴标签和图例优先使用中文；如果英文更符合数学或编程习惯，可以中英混用，但应保持一致。
6. 参考文献条目可以保留原始英文书名、论文名和链接，但参考文献前后的说明文字应使用中文。
7. 如果引用英文教材或英文资料，应尽量用中文概括其作用，例如“本节关于样条插值的推导可参考 Atkinson 的数值分析教材”。
8. 不要把中文内容机械翻译成生硬表达。写作风格应接近中文课程讲义，要求清楚、自然、适合教学。
9. 后续新增章节也必须遵守本语言规范，除非用户明确要求使用英文。

## 4. 章节组织原则

每一章建议包含：

* chapter overview / README
* theory-oriented notebooks
* algorithm-oriented notebooks
* experiment notebooks
* reference notes

每一章至少应有一个统一入口，用来说明：

* 本章主题；
* 阅读顺序；
* 主要数学问题；
* 主要算法；
* notebook 列表；
* 如何运行；
* 本章参考资料。

## 5. Notebook 写作规范

Jupyter Notebook 是本仓库的主要载体。每个 notebook 应尽量包含：

1. 问题背景；
2. 数学定义；
3. 关键公式；
4. 算法思想；
5. Python 实现；
6. 数值实验；
7. 图像展示；
8. 方法局限；
9. 小结。

Notebook 中的代码应以教学清晰为主，不追求过度工程化。允许使用 `numpy`、`matplotlib`、`scipy`，但核心算法应尽量给出手写实现。`scipy` 可以用于对照、验证和展示标准库接口。

图像应清楚说明物理或数学含义，不要只画图不解释。Notebook 输出应适度，不要提交巨大的无意义输出。

## 6. Notebook 与 scripts 的关系

本仓库允许在 `scripts`、`src` 或类似代码目录中维护可复用算法函数，但 Notebook 不能只是调用这些函数。

Notebook 是本仓库的主要教学载体，因此每个核心 Notebook 必须包含：

1. 问题背景；
2. 数学定义；
3. 关键公式推导；
4. 算法设计思想；
5. 核心算法的教学式实现；
6. 数值实验；
7. 图像展示；
8. 结果分析；
9. 小结。

`scripts` 中的代码可以用于：

* 复用算法；
* 避免多个 notebook 重复同一段长代码；
* 提供更规范的函数实现；
* 支持实验、画图和验证。

但 Notebook 中仍必须展示核心算法逻辑。具体要求：

1. 对于本章核心算法，Notebook 中必须至少给出一次“教学式实现”或伪代码解释。
2. 可以在 Notebook 后半部分调用 `scripts` 中的正式函数，但不能只调用函数而不解释算法。
3. 如果 Notebook 调用 `scripts` 中的函数，应说明该函数对应的数学公式和算法步骤。
4. `scripts` 中的函数应服务于 Notebook，不应取代 Notebook 的理论讲解。
5. 对重要算法，应同时保留：
   * Notebook 中的讲解版实现；
   * `scripts` 中的可复用版实现。
6. 如果教学版实现和 `scripts` 版实现不同，应说明差异，例如教学版偏清晰，`scripts` 版偏复用和稳定。
7. 不要把 Notebook 写成简单的 API demo。
8. 不要把核心理论和算法细节全部藏在 `.py` 文件里。

## 7. 理论写作规范

理论部分应做到：

* 给出基本定义；
* 说明算法从哪里来；
* 给出关键公式推导；
* 解释误差来源；
* 说明适用条件；
* 说明失败情形；
* 说明与后续章节的联系。

对于重要公式，应优先使用 LaTeX。不要只写最终公式，需要写清楚公式的意义和它在算法中的作用。

## 8. 代码规范

代码应满足：

* 可读；
* 可运行；
* 适合教学；
* 函数命名清楚；
* 变量名尽量反映数学含义；
* 关键步骤有简短注释。

核心算法函数应尽量独立，例如：

* `lagrange_interpolate(...)`
* `newton_divided_differences(...)`
* `cubic_spline_coefficients(...)`
* `chebyshev_nodes(...)`

不要求做成复杂 Python package，除非现有仓库已经采用 package 结构。本仓库当前采用轻量 package 结构，公共算法放在 `src/py_sc/`，章节示例放在 `chapters/<chapter>/`。

## 9. 实验规范

每章应设计若干数值实验，用来说明：

* 算法是否正确；
* 误差如何变化；
* 参数如何影响结果；
* 方法在哪些情况下失败；
* 不同算法之间的差异。

实验应尽量配图，并给出文字解释。不要只输出数字表格。

## 10. 参考文献规范

每章应维护参考文献，可以使用 Markdown、BibTeX 或现有仓库中的参考文献格式。

参考文献应优先包括：

* 数值分析经典教材；
* 科学计算教材；
* Python 科学计算资料；
* 与该章主题直接相关的专业资料。

引用不要求过度完整，但要避免完全没有来源。

## 12. Git 规则

修改完成后必须执行：

```bash
git status
git diff --stat
```

如果可行，应运行必要的代码检查或 notebook 检查。

默认不要执行：

```bash
git push
```

如果用户在当前对话中明确要求“推送”“push”“同步到 GitHub”或“同步到云端”，则应以用户的当前明确指令为准，可以执行 `git push`。发生冲突时，用户当前明确指令优先于本文件中的默认不 push 规则。

每次执行 `git push` 时，必须给出简短说明。说明应包括将要推送的仓库、分支和主要内容；推送完成后，应简要说明是否成功以及远端分支状态。若推送失败，应简要说明失败原因和下一步处理。

不要使用 `sudo`。不要安装系统级依赖。

如果需要新增依赖，优先说明理由，并尽量只写入项目级依赖文件。

## 15. 长任务、ExecPlan、断点与恢复

对于预计需要多个阶段、多个章节或较长时间完成的任务，必须使用 ExecPlan。

长期任务开始前：

1. 阅读 `.agent/PLANS.md`；
2. 创建或读取对应的活动 ExecPlan；
3. 将活动 ExecPlan 作为任务状态的权威记录；
4. 不得只依赖当前对话、模型记忆或终端滚动输出保存进度。

活动 ExecPlan 必须持续记录当前任务、当前章节、里程碑、已完成内容、未验证修改、检查结果、已知问题、最后安全 Git commit 和下一项具体动作。

每完成一个原子工作单元后，应更新活动 ExecPlan，向 `.agent/RUN_LOG.md` 追加日志，运行相应最小检查，并在安全且不会混入用户无关修改时创建本地 checkpoint commit。

不得在没有写入断点的情况下同时展开多个章节。新会话恢复时，应先读取 AGENTS、`.agent/PLANS.md`、活动 ExecPlan、运行日志、Git 状态和最近提交，再从 ExecPlan 的“下一项具体动作”继续。不要执行 `git push`。

从第 7 章开始，新增章节末尾只保留一个“本章小结”或“小结”；数值实验应嵌入对应算法 Notebook，资料来源使用 Notebook 内联引用、链接、脚注或仓库统一参考文献文件，不再为新章节单独建立“实验内容”或“参考文献”章节导航。
